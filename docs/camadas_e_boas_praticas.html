4.3 Camada de Modelos (Models)

O que são classes POCO e por que foram usadas

POCO (Plain Old CLR Object) é uma classe C# simples, sem herança de frameworks nem dependência de banco ou interface gráfica. No Pharmastock, os models são apenas “recipientes” de dados: propriedades com get e set, eventualmente um método estático de validação leve (como em Produto.CodigoBarrasValido).

Foram escolhidas porque:

  • ficam fáceis de ler e de apresentar na documentação do PIM;
  • podem ser reutilizadas pelo console e pelo WinForms sem duplicar código;
  • no próximo semestre, a mesma estrutura pode ser mapeada para tabelas SQL ou Entity Framework sem reescrever as telas.

Apresentação das classes: Produto, Funcionario, Fornecedor

ClasseAtributos principaisTipos
Produto Id, NomeProduto, CodigoBarras, PrecoVenda, PrecoCusto, IdFornecedor, RequerReceita int, string, decimal, bool
Funcionario Id, Nome, CPF, Cargo, SenhaHash, DataAdmissao, Ativo, DataDemissao int, string, DateTime, bool, DateTime?
Fornecedor Id, NomeEmpresa, Cnpj, Telefone int, string

Outras entidades no mesmo padrão: Lote, Venda, ItemVenda.

Atributos, tipos de dados e encapsulamento com propriedades (get/set)

Cada campo do domínio é exposto como propriedade automática (public int Id { get; set; }). Isso é o encapsulamento básico em C#: o código externo lê e altera valores pelas propriedades, sem acessar variáveis internas diretamente. Tipos como decimal são usados em preços; DateTime? em data de demissão permite valor nulo quando o funcionário ainda está ativo.

Referência — classe Produto.cs completa:

using System.Globalization;

namespace estoque_farmacia.Models;

public class Produto
{
    public int Id { get; set; }
    public string NomeProduto { get; set; }
    public string CodigoBarras { get; set; } = string.Empty;
    public decimal PrecoVenda { get; set; }
    public decimal PrecoCusto { get; set; }
    public int IdFornecedor { get; set; }
    public bool RequerReceita { get; set; }

    public static bool CodigoBarrasValido(string? codigo)
    {
        if (string.IsNullOrWhiteSpace(codigo))
            return false;
        return int.TryParse(codigo.Trim(), NumberStyles.Integer,
            CultureInfo.InvariantCulture, out _);
    }
}

4.4 Persistência de dados (versão atual do projeto)

Observação: nesta entrega o sistema não utiliza Entity Framework Core, AppDbContext nem PostgreSQL em tempo de execução. Os scripts SQL em backend/database/ servem como modelagem de referência para evolução futura.

A persistência é feita pelos Services, que mantêm listas estáticas em memória (List<Produto>, List<Fornecedor>, etc.), protegidas com lock para evitar conflito em acessos simultâneos. Ao fechar o programa, os dados são perdidos — comportamento aceitável para o protótipo acadêmico deste semestre.

A validação cruzada (ex.: produto só é salvo se o IdFornecedor existir em FornecedorService) substitui, na prática, restrições que no banco seriam foreign keys.

4.5 Camada de Serviços (Services)

Padrão de projeto: Single Responsibility + dependências no construtor

Cada Service cuida de um agregado (produtos, fornecedores, vendas…). Isso segue o Single Responsibility Principle (SRP): uma classe, uma razão principal para mudar.

O ProdutoService recebe FornecedorService no construtor para validar o fornecedor sem duplicar a lista de fornecedores — composição simples de dependências (próximo passo seria um container de DI no Program.cs, previsto para o próximo semestre).

ProdutoService — métodos CRUD

MétodoFunção
Salvar(Produto)Valida dados, verifica fornecedor e código de barras duplicado; gera Id e adiciona à lista
ListarTodos()Retorna cópia da lista de produtos
BuscarPorId(int)Localiza um produto pelo Id
Remover(int)Remove produto da lista em memória

Propriedade UltimoErro

Quando uma operação falha, o service preenche UltimoErro com mensagem legível e retorna false. A UI (console ou WinForms) exibe essa mensagem sem precisar capturar exceção em todo botão — reduz acoplamento e mantém o fluxo previsível para o usuário.

Referência — ProdutoService.cs (estrutura principal):

4.6 Implementação completa de um caso de uso — Cadastro de Produto

Justificativa da escolha

O cadastro de produto é central no domínio da farmácia: conecta fornecedor (validação cruzada), alimenta lotes e vendas, e existe nas duas interfaces (console e WinForms). Por isso é um bom exemplo de fluxo ponta a ponta sem ser o caso mais complexo (venda + estoque).

Fluxo completo

Usuário preenche formulário (ProdutoForm ou ProdutoUI no console) ↓ UI monta objeto Produto com dados digitados ↓ ProdutoService.Salvar(produto) — validações e gravação na lista em memória ↓ Retorno true → mensagem de sucesso e atualização da grade/lista

Regra de negócio no Service, não só no botão

No WinForms, o botão Salvar faz validações de interface (campos vazios, formato de número). Regras de negócio (fornecedor inexistente, código de barras duplicado) ficam no ProdutoService.Salvar. O trecho final do clique resume a chamada ao service: 

if (_service.Salvar(novo))
{
    MessageBox.Show("Produto salvo.", "Sucesso", ...);
    LimparCampos();
    CarregarDados();
}
else
{
    MessageBox.Show(..., _service.UltimoErro, ...);
}

4.7 Composição de dependências (Program.cs)

O que é injeção de dependência e o que o projeto faz hoje

Injeção de dependência (DI) significa que uma classe recebe suas colaborações prontas (por construtor), em vez de criar new ProdutoService() dentro de cada tela. Isso reduz acoplamento e facilita testes.

Nesta versão, a “montagem” é manual no Program.cs (console): instancia-se FornecedorService, depois ProdutoService(fornecedorService), depois LoteService, Estoque, VendaService, e por fim as classes UI recebem os services já configurados. No WinForms, o MenuForm faz o mesmo no construtor.

Não há Microsoft.Extensions.DependencyInjection nem container nesta entrega — evolução planejada para quando houver persistência em banco.

Referência — Program.cs:

var fornecedorService = new FornecedorService();
var produtoService = new ProdutoService(fornecedorService);
var loteService = new LoteService(produtoService);
var estoque = new Estoque(loteService);
var vendaService = new VendaService(produtoService, estoque);

var menu = new Menu(
    new FuncionarioUI(funcionarioService),
    new ProdutoUI(produtoService),
    new FornecedorUI(fornecedorService),
    new LoteUI(loteService),
    new VendaUI(produtoService, vendaService, estoque));

4.8 Boas práticas de OO aplicadas

Princípio / práticaOnde aparece no Pharmastock
Encapsulamento Models com propriedades get/set; listas dos services são private static; UltimoErro só leitura externa (get; private set)
Single Responsibility ProdutoService só produtos; FornecedorService só fornecedores; telas só interação
Separação de responsabilidades Pastas Models / Services / UI (e Forms no WinForms); regra de negócio fora do MessageBox
Composição ProdutoService usa FornecedorService; VendaService usa Estoque e ProdutoService
Reuso WinForms referencia os mesmos .cs de Models/Services via Link no .csproj

O que foi evitado

  • God Class — não há uma única classe com todos os cadastros; há UI e Service por módulo.
  • Lógica pesada no evento do botão — validação de negócio concentrada no Service (o botão valida só entrada do usuário).
  • UI acessando lista interna do service — a UI chama métodos públicos (Salvar, ListarTodos).
  • Acesso direto ao banco pela UI — não há banco na versão atual; quando houver, o acesso ficará nos Services.

Interfaces (IProdutoService, etc.) ainda não foram introduzidas — simplificação intencional para o 3º semestre.
