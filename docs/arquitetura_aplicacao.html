Arquitetura da Aplicação

O Pharmastock (Sistema de Estoque para Farmácia) foi organizado em três camadas lógicas — Models, Services e UI — no projeto principal estoque_farmacia. Essa divisão segue um padrão simples de aplicação em camadas, adequado ao estágio do curso: cada parte do código tem um papel claro, o que facilita manutenção, trabalho em equipe e evolução futura (por exemplo, persistência em banco de dados no próximo semestre).

O sistema possui duas interfaces de usuário:

  • Console — menus em texto na pasta UI/ do projeto estoque_farmacia;
  • Windows Forms — telas gráficas no projeto estoque_farmacia_winforms, na pasta Forms/.

Ambas reutilizam os mesmos Models e Services do projeto base. No WinForms, os arquivos compartilhados são referenciados via Link no .csproj, evitando duplicar regras de negócio.

Visão geral das três camadas

1. Camada Models (estoque_farmacia.Models)

Representa as entidades do domínio — os “dados” do sistema, sem lógica de tela ou de persistência complexa.

ClassePapel resumido
FuncionarioDados do funcionário (nome, CPF, cargo, senha hash)
ProdutoMedicamento/produto (preços, código de barras, etc.)
FornecedorEmpresa fornecedora
LoteLote de produto (validade, quantidade)
VendaRegistro de venda (data, totais, itens)
ItemVendaItem individual dentro de uma venda

As classes são principalmente propriedades (get/set), espelhando tabelas que o grupo modelou em SQL na pasta backend/database.

2. Camada Services (estoque_farmacia.Services)

Concentra a lógica de negócio e o armazenamento em memória (listas estáticas com lock para acesso seguro).

Serviço / classeResponsabilidade principal
FuncionarioServiceCRUD de funcionários, validação de CPF duplicado
ProdutoServiceCRUD de produtos; valida fornecedor e código de barras
FornecedorServiceCRUD de fornecedores
LoteServiceCRUD de lotes; valida se o produto existe
EstoqueConsulta e baixa de quantidade nos lotes
VendaServiceFinaliza venda, valida estoque, registra venda e baixa lotes
SenhaHasherGera hash SHA-256 da senha do funcionário

A UI não manipula listas diretamente: ela chama métodos como Salvar, ListarTodos, Finalizar, e lê UltimoErro quando algo falha.

3. Camada UI

Console (estoque_farmacia.UI) — interação via Console.ReadLine, menus e mensagens coloridas:

  • Menu — login e menu principal
  • FuncionarioUI, ProdutoUI, FornecedorUI, LoteUI, VendaUI — fluxos por módulo

Windows Forms (estoque_farmacia_winforms) — equivalente gráfico:

  • Forms/ — LoginForm, MenuForm, ProdutoForm, FuncionarioForm, etc.
  • UIHelper — cores e estilo visual comum às telas

O ponto de entrada do console é Program.cs, que instancia os serviços e injeta nas classes UI. O WinForms faz o mesmo em MenuForm, mantendo uma única fonte de dados (listas estáticas nos services).

Justificativa da separação de responsabilidades

Por que existe a camada Models?
Para definir o que o sistema manipula (funcionário, produto, venda), independentemente de ser exibido no console, em formulário Windows ou, no futuro, em API web. Se a estrutura de um produto mudar (novo campo), altera-se principalmente o model e os pontos que o utilizam, sem misturar isso com código de botão ou Console.WriteLine.

Por que existe a camada Services?
Para concentrar regras e validações em um só lugar: CPF duplicado, fornecedor inexistente ao cadastrar produto, estoque insuficiente na venda, baixa em lotes, hash de senha. A UI só pergunta dados ao usuário e chama o serviço; assim, console e WinForms não repetem a mesma lógica. Isso também prepara o projeto para trocar memória por banco depois — a mudança tende a ficar nos services, não em todas as telas.

Por que existe a camada UI?
Porque apresentação e negócio são preocupações diferentes. O ProdutoUI lê nome e preço no terminal; o ProdutoForm usa TextBox e DataGridView — mas ambos chamam ProdutoService.Salvar. Trocar a interface (só console, só WinForms, ou as duas) não obriga reescrever validação de estoque ou de fornecedor.

Fluxo resumido (dependência entre camadas):

UI (Console / WinForms)
        ↓ chama métodos
Services (regras + dados em memória)
        ↓ usa objetos
Models (entidades)

A UI não deve depender de detalhes internos das listas nos services; os services não devem conhecer MessageBox ou cores do console (embora hoje alguns services ainda usem Console.WriteLine para mensagens — ponto de melhoria futura).

Benefícios para o projeto acadêmico: código legível para a banca, divisão de tarefas entre integrantes (um cuida de models/services, outro de forms/UI), alinhamento com o modelo entidade-relacional documentado em docs/ e backend/database/, e base para evoluções planejadas (persistência, login por funcionário cadastrado, API, etc.) sem refatoração total.
