4.X Inicialização da aplicação (Program.cs)

O arquivo Program.cs é o ponto de entrada da versão em console do Pharmastock. Quando o usuário executa o projeto estoque_farmacia, o .NET começa por esse arquivo: antes de qualquer menu aparecer, o sistema precisa criar os serviços (camada de regras de negócio) e conectá-los às telas em texto (camada UI). Tudo isso é feito de forma explícita nas primeiras linhas do programa.

Montagem dos serviços

Na sequência abaixo, cada serviço é instanciado respeitando as dependências entre eles:

  1. FuncionarioService e FornecedorService — não dependem de outros serviços; são criados primeiro.
  2. ProdutoService(fornecedorService) — precisa do fornecedor para validar se o IdFornecedor existe ao cadastrar produto.
  3. LoteService(produtoService) — só permite lote para produto já cadastrado.
  4. Estoque(loteService) — calcula e baixa quantidade a partir dos lotes.
  5. VendaService(produtoService, estoque) — finaliza vendas checando produto e estoque.

Essa ordem evita erro de lógica (por exemplo, cadastrar produto com fornecedor inexistente ou vender sem controle de lote) e mantém uma única “fonte da verdade” em memória: as listas estáticas dentro dos services são compartilhadas por todas as telas abertas na mesma execução.

Ligação com a camada UI

Em seguida, o código cria o Menu principal e injeta manualmente cada interface de usuário, já com o service correto:

  • FuncionarioUI → FuncionarioService
  • ProdutoUI → ProdutoService
  • FornecedorUI → FornecedorService
  • LoteUI → LoteService
  • VendaUI → ProdutoService, VendaService e Estoque

Assim, quando o usuário escolhe a opção “Controle de Produto” ou “Controle de Venda” no menu, a tela correspondente não cria um service novo: ela usa o mesmo objeto criado no Program.cs, garantindo que cadastro feito em um módulo apareça nos outros (por exemplo, produto cadastrado pode ser usado em lote e venda).

Fluxo de execução

Depois da montagem, o programa:

  1. Chama menu.ValidarLogin() — até três tentativas com usuário/senha fixos (admin / 123).
  2. Se o login for aceito, executa menu.ProcessarMenu() — loop com as opções 1 a 5 (produto, funcionário, venda, fornecedor, lote) e saída com opção 7.
  3. Se o login falhar, exibe mensagem e encerra.
  4. Um bloco try/catch envolve a inicialização para capturar falhas inesperadas e exibir erro sem derrubar o processo sem aviso.
Program.cs → cria Services (cadeia de dependências) → cria Menu + UIs (cada UI recebe seu Service) → ValidarLogin() ├─ ok → ProcessarMenu() (usuário navega nos módulos) └─ falha → encerra com mensagem

Referência — código do Program.cs

using estoque_farmacia.UI;
using estoque_farmacia.Services;

var funcionarioService = new FuncionarioService();
var fornecedorService = new FornecedorService();
var produtoService = new ProdutoService(fornecedorService);
var loteService = new LoteService(produtoService);
var estoque = new Estoque(loteService);
var vendaService = new VendaService(produtoService, estoque);

var menu = new Menu(
    new FuncionarioUI(funcionarioService),
    new ProdutoUI(produtoService),
    new FornecedorUI(fornecedorService),
    new LoteUI(loteService),
    new VendaUI(produtoService, vendaService, estoque));

try
{
    if (menu.ValidarLogin())
        menu.ProcessarMenu();
    else
        Console.WriteLine("Sistema encerrado por não autorização.");
}
catch (Exception ex)
{
    Console.WriteLine($"ERRO ao inicializar aplicação: {ex.Message}");
}

Relação com o WinForms

Na versão gráfica, o papel equivalente fica dividido: o Program.cs do WinForms apenas inicia o LoginForm; a montagem dos services ocorre no construtor do MenuForm, com a mesma ideia (fornecedor → produto → lote → estoque → venda). Nos dois casos trata-se de composição manual de dependências, sem container de injeção de dependência — solução simples, adequada ao escopo acadêmico do projeto.
